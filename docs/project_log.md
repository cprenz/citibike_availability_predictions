## 2.05 Logistic Regression — tol/OOM tuning journey and chart list — 2026-06-24/25

Archived from CLAUDE.md Phase 3 checklist, the 2.05 entry's "DECISIONS AND FIXES" /
"COMPARISON RESULTS" / "CHARTS" blocks. Original block, verbatim:

> ORIGINAL STATE (for reference): COMPARISON_MODE=False, tol=1e-2, full 6-horizon run.
> Run in VSCode notebook mode (not PowerShell) so chart outputs embed in the .ipynb.
> DECISIONS AND FIXES (all 2026-06-24):
> - NO Optuna C search — dropped. At 12M rows L2 is inert; unregularized
>   (penalty=None) gives essentially the same model with less runtime.
> - statsmodels Logit inference table on 300k-row subsample (not full 12M):
>   avoids Hessian singularity from exact collinearities. QR drops 5 columns:
>   season_fall, station_role_source, num_bikes_disabled_was_null, ebike_ratio,
>   is_holiday. Top drivers: net_flow_3hr (z=48.8), fill_ratio (z=34.9),
>   cumulative_expected_net_flow_12hr (z=26.2).
> - DECISION: NO class_weight="balanced" — balancing distorts calibrated
>   probabilities; AUC is threshold-independent so unaffected.
> - tol=1e-3 tested first — DID NOT WORK. Saga hit max_iter=200 on every fold
>   without triggering tol (gradient decayed to ~0.002, never reached 0.001).
> - tol=1e-2 tested — WORKS. Saga converges at ~40 epochs per fit. Runtime
>   dropped from 2.3 hrs → 33 min for the 1hr horizon. Holdout AUC 0.9382
>   (BETTER than the 0.9331 from max_iter=200 — early stopping acts as implicit
>   regularization for saga, preventing overfitting to the 2019/2021 era).
> - OOM FIX: CV training set capped at 6M rows in run_cv(). Final fit uses full
>   12M rows unaffected. Fold 5 at full size hit ~6.3GB dense float64 matrix
>   and was silently OOM-killed on Windows.
> COMPARISON RESULTS (1hr horizon, tol=1e-2 vs max_iter=200):
>   tol=1e-2:     CV AUC 0.9099±0.0144  holdout AUC 0.9382  Brier 0.0477  33min
>   max_iter=200: CV AUC 0.9147±0.0177  holdout AUC 0.9331  Brier 0.0491  2.3hr
>   tol=1e-2 WINS on holdout AUC, Brier, and runtime. Validated.
> CHARTS (all saved to reports/figures/):
>   2.05_logit_coefficients_{label}.png — top 15 coefficient chart PER HORIZON
>     (standalone 3hr section uses coef_df from statsmodels; remaining 5 horizons
>     generated by a loop after the 3hr section using plot_coef_chart() helper —
>     300k subsample + QR collinearity drop + statsmodels fit per horizon)
>   2.05_roc_curves_by_horizon.png — all 6 ROC curves overlaid
>   2.05_cv_vs_holdout_auc.png — CV vs holdout AUC per horizon
>   2.05_holdout_brier_by_horizon.png — Brier score per horizon
>   2.05_pr_curves_by_horizon.png — precision-recall curves overlaid across all 6
>     horizons (AP + base rate in legend). PR curves show empty-station detection

## Hypothesis test 1.03 — rider-explained vs unexplained decomposition query — 2026-06-27

Archived from CLAUDE.md Phase 3 checklist, the 1.03 entry's "CROSS-REFERENCE RESEARCH"
block (the interpretation summary is retained, condensed, in CLAUDE.md). Original
block, verbatim:

> CROSS-REFERENCE RESEARCH (2026-06-27, NOT in notebook): decomposed fill_ratio_change_1hr
> into rider-explained vs unexplained components by joining station_hourly_flow for the
> same hours (May 2026 only -- June trips not yet published). Rider-explained =
> (arrivals - departures) / capacity. Unexplained = observed - rider_explained.
> Full 24-hour query results (hour_of_day = UTC):
>   morning (UTC 10-12): observed -0.0143, rider_explained -0.0009, unexplained -0.0134,
>     avg_dep 1.716, avg_arr 1.717
>   evening (UTC 21-23): observed +0.0091, rider_explained +0.0113, unexplained -0.0023,
>     avg_dep 1.213, avg_arr 1.484
> INTERPRETATION: The two windows have DIFFERENT root causes, not a symmetric rider flow.
> (a) Morning drop is NOT rider-driven: departures (1.716) ≈ arrivals (1.717), so riders
>     explain almost nothing (rider_explained -0.0009 vs observed -0.0143). The drop is
>     overwhelmingly unexplained (-0.0134), implicating rebalancing trucks removing bikes
>     from stations in the morning.
> (b) Evening gain IS rider-driven: arrivals (1.484) > departures (1.213), rider_explained
>     +0.0113 accounts for nearly the full observed gain. Rebalancing slightly negative
>     (-0.0023). Riders genuinely return more bikes than they take in the evening.
> The 24-hour table shows avg_available peaks at 3-5am EDT (~14.76 bikes) then declines
> all day to a trough at 5pm EDT (UTC 21, ~13.34 bikes) and recovers overnight. This is
> the "bikes in transit" effect: bikes being ridden are docked nowhere, so total docked
> availability drops during peak ride hours and recovers overnight as bikes settle.
> Disabled bikes go in the OPPOSITE direction (lowest in morning, highest at night) and
> do NOT explain the daytime drain.
> TIMEZONE CAVEAT (discovered during this research): station_hourly_flow.hour is stored in
> local America/New_York time, while training_features.timestamp is true UTC. The join
> `f.hour = tf."timestamp"` in notebook 1.01 is therefore misaligned by ~4 hours (EDT).
> This affects the trip-count numbers joined in 1.01 but NOT the fill_ratio_change_1hr
> values or the 1.03 statistical test itself (which uses only training_features columns).
> FIXED (2026-06-27): 1.01 timezone join corrected and re-run — see 1.01 entry above.
> 1.03 updated with a decomposition section (rider-explained vs unexplained) using the
> same timezone-correct join, answering why morning is negative and evening is positive.

## Phase 3.5 — gitignored models/ artifact recovery workaround — 2026-06-30

Archived from CLAUDE.md Phase 3.5 checklist, the "GOTCHA HIT + WORKAROUND PATTERN"
entry (a condensed version of the reusable pattern is retained in CLAUDE.md).
Original block, verbatim:

> GOTCHA HIT + WORKAROUND PATTERN (2026-06-30): `models/` is gitignored, so
> .joblib artifacts do NOT persist across sessions/restarts — they only exist on
> whatever machine last ran the training notebook. 2.08b crashed with
> `FileNotFoundError: models/linear_regression_60min.joblib` because the 2.04b
> artifacts referenced in the Phase 3.5 design (from 2026-06-25) were gone by
> 2026-06-30. Do NOT default to retraining 2.04b (~78 min) to fix this — the
> user has explicitly said they don't want to retrain when avoidable. The fix:
> `reports/figures/2.04b_horizon_metrics.csv` (NOT gitignored, survives) already
> contains the `pi_lower_offset` / `pi_upper_offset` / per-horizon columns that
> 2.08b actually reads before it overwrites the artifact — the rest of the old
> artifact (the fitted pipeline) is discarded by 2.08b anyway, never used for
> inference. So: build a STUB artifact per horizon (a dummy unfitted `Pipeline`
> object + the real PI offsets pulled from the CSV + `pi_level: 0.95`), `joblib.dump`
> it to `models/linear_regression_{h}min.joblib`, then run 2.08b normally — it
> reads the offsets, discards the stub pipeline, and overwrites with the real
> full-data fit. Verified afterward: the recovered offsets in the final 2.08b
> artifacts exactly match the original 2.04b CSV values. General pattern for
> this project: when a downstream notebook needs a prior `models/*.joblib`
> artifact that's gone, check whether the metrics it actually consumes (not the
> full fitted model) survive in a `reports/figures/*.csv` log before retraining —
> gitignored artifacts disappear, committed CSVs do not.
